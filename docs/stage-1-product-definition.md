# Этап 1: Product Definition и фиксация концепции (4–6 недель)

Источник этапа: `/Users/denecer/Yandex.Disk.localized/Научные материалы/Патентные заявки/Урофлоуметр/deep-research-report.md`

Продолжение развития: `docs/stage-2-fusion-development.md`.

## Цель этапа

Зафиксировать границы продукта и технически реализуемый MVP так, чтобы команда могла перейти к инженерной реализации и pilot-валидации без смены базовых допущений.

## Что считаем MVP в этом проекте

Основной режим MVP:
- `Volume-based video uroflowmetry` (видео измеримого объёма, а не "чисто по струе").
- Выход: `V(t)` и `Q(t)=dV/dt` + производные метрики:
  `Qmax`, `Qavg`, `Voiding Time`, `Flow Time`, `Time to Qmax`, `Voided Volume`, `Intermittency`.

Дополнительный режим (R&D, не часть MVP-claims):
- `Jet-based analysis` (угол, распыление, визуальные симптомы струи).

## Границы клинических утверждений

В рамках MVP нельзя заявлять измерение:
- детрузорных/уретральных давлений;
- остаточной мочи (PVR) без дополнительного датчика/УЗИ;
- диагностических заключений без клинической валидации.

## Минимальные требования к измерению (Stage 1 baseline)

1. Интерпретируемые измерения ориентировать на `voided volume > 150 ml`.
2. Повторяемость: предусмотреть как минимум 2 повторных измерения при сомнительном результате.
3. Видео-пайплайн MVP ориентировать на стабильный захват и калибровку масштаба.
4. В отчёте обязательно хранить quality flags (калибровка, стабильность, полнота акта, confidence).

## Data Governance и privacy-by-design (обязательные решения на этапе 1)

1. Режим по умолчанию: обработка на устройстве и хранение производных метрик вместо сырого видео.
2. Сырой видеоархив только по явному согласию пользователя и с ограниченным TTL.
3. Базовая схема de-identification: ROI-only обработка и исключение лишнего фона.
4. Подготовить DPIA checklist до pilot-сбора данных.

## Регуляторная рамка (предпроектная фиксация)

1. Зафиксировать intended use и non-claims для R&D версии.
2. Подготовить план перехода к clinical-ready:
   ISO 13485, IEC 62304, ISO 14971, IEC 62366-1, IEC 82304-1.
3. Вести FTO-backlog по патентному ландшафту видео-урофлоуметрии.

## Deliverables этапа 1

1. Product definition:
   - intended use;
   - target users;
   - claims / non-claims;
   - ограничения MVP.
2. Measurement spec:
   - формулы `V(t), Q(t)`;
   - определение quality flags;
   - критерии приемки точности.
3. Data & privacy pack:
   - data-flow схема;
   - retention policy;
   - consent text draft;
   - DPIA checklist.
4. Validation draft:
   - pilot протокол;
   - метод сравнения с эталонным урофлоуметром;
   - метрики Bland-Altman / MAE / ICC.
5. Engineering backlog:
   - задачи Phase 2+ с оценкой и зависимостями.

## Definition of Done для этапа 1

- [ ] Утверждён документ Intended Use + границы claims.
- [ ] Утверждён выбор базового режима MVP (`volume-based`).
- [ ] Утверждена спецификация метрик и quality flags.
- [ ] Подготовлены шаблоны pilot-валидации и критерии сравнения с эталоном.
- [ ] Подготовлен privacy пакет (retention, consent, DPIA checklist).
- [ ] Сформирован FTO backlog и список проверок перед clinical-ready фазой.

## Риски этапа 1 и меры снижения

| Риск | Влияние | Мера снижения |
|---|---|---|
| Размытые claims | Невозможность корректной валидации/регуляторики | Жёстко зафиксировать non-claims и intended use |
| Нестабильная геометрия съёмки | Ошибки `V(t)` и `Q(t)` | ROI + маркер масштаба + quality flags |
| Сырой видео-поток без ограничений | Privacy/юридические риски | On-device по умолчанию, TTL, явное согласие |
| Слишком ранняя ставка на jet-based режим | Рост сложности и сроков | Разделить на MVP (`volume-based`) и R&D ветку |
